# 0001: Start with a private Databricks App deployed through a Bundle

- Status: Accepted
- Date: 2026-07-15
- Decision owner: Product owner
- Review state: Baseline 0.20 P0 acceptance candidate; immutable verdicts and resolutions are recorded under `docs/reviews/p0-planning-baseline/`

> **v0.3.0 qualification note:** The accepted product shape remains a private App plus Bundle, but the first supported release is narrower than the later architecture explored in this decision record. Its authoritative [support contract](../releases/v0.3.0-support-contract.md) supports `COMBINED_ROLE` only and uses one temporary serverless Python-wheel bootstrap Job running as the schema-owner administrator. The native Statement Execution mutation registry, separated duties, runtime-trust ledger, controlled actions, system enrichment, upgrade, rollback, and Marketplace are not invoked or supported by v0.3.0.

## Context

The product must make dbt Core job observability easy to install and operate in a highly regulated environment using Databricks on Azure. Required telemetry must not leave the customer's Databricks account. The initial test workspace is personally funded, so the design must not depend on continuously running test resources.

The product needs two distinct capabilities:

1. Deterministic collection and normalization of dbt and Lakeflow execution evidence.
2. A clear operator experience for installation, health, run history, investigation, retention, cost, and removal.

## Decision

Build a private Databricks App and deploy the App plus product-owned Databricks resources using a plain-YAML Databricks Declarative Automation Bundle.

The App is a stateless UI and API control plane. It does not run dbt in its own process and does not continuously poll Jobs. An observed Lakeflow Job runs dbt under its existing job-level run-as principal. Its downstream `ALL_DONE` Run Job task may only invoke a separately defined collector Job, which runs under a narrower collector principal. Unity Catalog managed tables store normalized evidence; a separately permissioned managed Volume may store restricted closed diagnostic evidence under an explicit retention policy.

CLI 1.9.0 Bundles have no generic managed Delta-table or view resource, and an App Unity Catalog binding requires an existing object. The customer therefore supplies one dedicated schema as a typed, independently verified input. The Bundle itself neither creates nor adopts that schema and contains no authoritative schema `grants` list. A saved Direct runtime plan manages only supported App, collector, and optional role/enrichment resources, their ACLs, and verified existing-object bindings; it never mutates schema privileges, writes a receipt, runs DDL, or claims that Direct created a table or view. A separately surfaced collector bootstrap entry point may create the fixed product objects in `BOOTSTRAP_ALLOWED`; ordinary sweeps use `RUNTIME_DML_ONLY`. Personal/test self-bootstrap and attended regulated bootstrap are distinct journeys with different authority disclosures.

One protected, deterministic `migration-envelope.json` covers the complete privilege/data change. It names only product-recorded additive parent-use grants when required, fixed DDL and idempotent data operations, exact object owners and non-App grants, verification, and the exact temporary migration-ledger `SELECT`/`MODIFY` grant to a named customer-managed operator group plus matching revoke. The attended Unity Catalog operator approves the canonical operations, and the signed wrapper re-renders and executes the complete envelope in memory through the Statement Execution API under that operator's OAuth U2M identity. It never executes SQL supplied by the envelope, Direct plan, user, App, AI, or Job parameter. There is no privileged migration Job or migration run-as service principal.

The v1 regulated separated-duties topology is exactly one customer-controlled supported workstation with distinct managed OS accounts for two people: a deployment/seal-verifier administrator and a UC operator. The verifier is both workspace administrator and account administrator and performs the native service-principal-roster review; a third enterprise account reviewer is unsupported in v1. Each person uses their own complete OAuth profile and native secure store in their own account. A system-owned local spool carries an auto-discovered, signed, expiring, replay-resistant handoff capsule containing only typed nonsecret state and digests; it never carries a token, raw identity, executable SQL, or another person's checkpoint. Both approved groups have direct `CAN_MONITOR` on one customer-supplied installer-only SQL warehouse, accepting that monitors can see full query text and identities there. The verifier also has effective `CAN_MANAGE` on the warehouse and is disclosed as a warehouse-management trusted root; another named manager is the procedural escalation owner, not the sole technically capable manager. The UC account owns fixed-SQL schema/data execution, bounded polling, idempotent recovery, the exact group-grant revoke, and the pre-revoke data attestation. The deployment account resumes only after the UC operator releases the interactive OS session, the matching remote attestation is readable, Query History proves every operation terminal, and the different verifier recomputes complete live authority. Separated mode therefore separates attended UC schema/data mutation from later verification, but it does not claim an independent SP-role reviewer: it records `human_actor_count=2`, `data_mutation_observer_separation=true`, and `independent_sp_role_reviewer=false`. Same-OS-account separation, ad hoc separate-workstation transfer, and a third-reviewer topology are unsupported; a personal combined-role path is supported but never described as independently separated.

The personal `COMBINED_ROLE` path uses one accountable account administrator, one supported OS account, and one actor-owned U2M profile that proves the union of deployment, UC-operator, seal-verifier, and SP-role-review prerequisites. Before mutation it records `human_actor_count=1`, `independent_human_separation=false`, `data_mutation_observer_separation=false`, and `independent_sp_role_reviewer=false`. It preserves recovery audit, exact envelope approval, durable cleanup readiness, bounded fixed Statement Execution, exact revoke, live data/owner/grant classification, the remote pending attestation, staged and stopped App deployment, native service-principal-roster review, and explicit same-deployment start. It emits no capsule, OS-account-release proof, or deployer-return receipt. A personal actor without account-admin authority is unsupported for this one-human route. The selected mode becomes immutable after the first mutation; same-account `SEPARATED_DUTIES` fails rather than silently becoming combined.

Every mutating statement uses one request with `wait_timeout=50s`, `on_wait_timeout=CANCEL`, bounded client timeouts, and no blind retry after an indeterminate response. Before submission, a completed read-only query marker durably exposes one signed closed cleanup locator in Query History: installation/generation, operation UUID, exact operator group/securable/privilege/action, and envelope/statement digests. Only the submitting user fetches Statement Execution results. A replacement uses the GA Query History API/UI on the dedicated warehouse, treats cancellation as nonterminal until terminal state is observed, and reconstructs Unity Catalog post-state; the Preview query-history system table and query tags are not dependencies. Ambiguous lookup or a query still nonterminal after ten minutes escalates to the named warehouse manager and blocks all mutation.

Every wrapper start or resume also audits all product objects and owners and complete GA-visible data authority, including direct and inherited grants, implied `ALL PRIVILEGES`, current effective DML, `MANAGE`/ownership self-grant capability, caller visibility, and pagination. The administrator runs `SHOW GROUPS WITH USER`/`WITH GROUP` for the named migration actors and group to classify their direct and indirect current-DML paths. Every granted group principal is itself a governed trusted root; dbtobsb does not claim complete membership enumeration or continuous member-drift detection. An incomplete operation or residual product-created temporary ledger authority enters cleanup only: no product compute or unrelated mutation starts until Query History is terminal, the exact targeted revoke is resolved, and live authority is freshly classified. Unexpected customer/group/inherited authority requires its owning customer administrator and never triggers a broad revoke.

The customer-security-owned Delta ledger holds one closed, digest-bound, idempotent `DATA_APPLIED_PENDING_REVOKE` row. Its fixed write derives actor and time from the platform, binds the installation/generation, prior-attestation hash, envelope/source/build/object/intended-authority digests and completed prior Statement Execution IDs, and rejects a conflicting row. It cannot contain its own statement ID or attest a revoke that occurs later. The deployment operator has persistent `SELECT` only on a sanitized current-attestation view; every runtime principal has no ledger DML, DDL, ownership, or `MANAGE`.

Runtime data-contract unlock is instead one composite point-in-time acceptance tuple: matching remote attestation; exact product-created direct ledger pair absent; complete observed objects, owners, and non-App grants; direct/indirect group paths for the named migration actors/group; each granted group principal as a trusted root; current effective `SELECT`/`MODIFY` paths; ownership/`MANAGE` self-grant paths; named trusted residuals; no indeterminate statement; verifier/time; mode gate; and a canonical seal digest. Current effective DML blocks; bounded trusted self-grant capability is reported rather than mislabeled as data access. The tuple is recomputed before each runtime plan and trust generation. A capsule, checkpoint, copied JSON, Delta row alone, or Jobs output cannot unlock runtime.

Account/workspace/metastore administrators, customer schema/product-object and attestation-security owners, every granted group principal, the object-scoped seal-verifier group, Bundle deployment and product/App administrators, product Job owners/`CAN_MANAGE` principals, and every accepted Service Principal User/Manager principal are explicit trusted roots. The seal-verifier group requires complete `SHOW GRANTS` visibility through exact object/container ownership or `MANAGE`; workspace-admin status alone is not represented as Unity Catalog visibility. The GA path cannot machine-read the complete service-principal role rule set, so an account administrator must compare each native account-console Permissions roster with the signed expected roster and create a separately labeled point-in-time administrative attestation. `RestrictWorkspaceAdmins` is defense in depth, not administrator-resistant custody. A customer that requires protection from these roots or current machine-verifiable service-principal-role drift fails readiness rather than receiving a false platform-enforced separation claim.

Direct is only the resource-deployment plane. It can create or update approved App/Job resources, ACLs, schedules, run-as references, and bindings, but executes no SQL and holds no UC DDL/DML. Deployed runtime principals intentionally write only their enumerated data sets: collector evidence, optional enrichment snapshots, or P6 action/fence state. A signed runtime-trust manifest binds an attended, timestamped machine observation of GA-visible code, task/configuration, run-as, ACL, binding, and UC-grant state to the same verifier-administrator's separately labeled native service-principal-roster attestation.

One restricted customer-security-owned Delta event table, `runtime_trust_ledger`, and sanitized `runtime_trust_status_v` view are the durable authority. The verifier group has persistent `SELECT` and `MODIFY` only on that control ledger and is an explicit trusted root; no runtime principal can write or promote trust. One physical Delta row is one logical event. Each generation freezes a canonical sorted, unique typed component array containing `BASE_OBSERVABILITY` and only the installed `SYSTEM_ENRICHMENT` or `CONTROLLED_ACTIONS` components. Fixed, marked, idempotent Statement Execution appends immutable-by-contract `MANIFEST_REGISTERED`, `TRUST_CANDIDATE`, `SNAPSHOT_ACCEPTED`, or `SNAPSHOT_INVALIDATED` rows. The product plan freezes exhaustive DDL, exact per-event required/null rules, versioned canonical objects, literal domains, null representation, non-circular RFC 8785/SHA-256 dependency order, and cross-language integer encoding: Delta values remain range-checked numeric columns while every canonical JSON integral property is an unsigned decimal string. The server supplies `statement_evaluated_at` from one statement-scoped `current_timestamp()` at query-evaluation start; it is not called a Delta commit time.

A candidate row holds the complete pre-start machine observation with `STOPPED`, the selected deployment, zero pending deployments, and a stable graph digest. An acceptance row repeats that pre-start digest and every stable fact, then adds a distinct post-start observation with `ACTIVE`, the same selected deployment, zero pending deployments, and the same stable graph. Lifecycle state is excluded from the immutable deployment inventory. Deployment-bearing generations prove exactly one new record; an `UNCHANGED_REFRESH` stops and restarts the already accepted deployment with byte-equal before/after inventories and a zero-sized difference. The post-start statement time is the machine freshness anchor. A newly completed roster comparison is the candidate's original self-anchor. An early unchanged refresh can carry that exact original event ID and roster digest only if the original candidate belongs to one complete accepted, never-invalidated, conflict-free source generation; it may not reference an intermediate reuse candidate. The current view revalidates that entire source chain on every query, so later invalidation/deletion/conflict makes the dependent current generation unverified. Expiry always uses the original roster statement time and can never move forward without a new native review.

The status view reduces only the installation-wide latest generation. Positive state requires exactly one valid registration, candidate, and acceptance row; exact predecessor links; zero invalidations or unknown operations; one pre-start and one post-start observation with the sole `STOPPED` to `ACTIVE` transition and identical stable graph; one independently valid original roster source; one matching expected/observed component set; exactly one reconciled deployment; matching manifest/account/workspace/authority/roster digests; valid canonical identifiers; and no duplicate event, ID, payload, or snapshot. Invalidation has precedence over acceptance, and any newer registration makes every older snapshot non-current. The view returns one installation summary with the complete typed component array and derives `ACCEPTED`, `STALE`, or `UNVERIFIED` using server query-evaluation `current_timestamp()`. App and bounded writers receive `SELECT` only on the view and never promote trust. Optional client signer fingerprints/signatures may be retained as non-authoritative audit metadata, but neither the view nor a runtime gate uses them.

The collector freezes the first status-view snapshot/generation/state/query-evaluation time on the immutable `dbt_artifact_registry` AttemptKey root. It reuses that tuple through retries and partial writes and labels it **Trust observed by collector**, separate from current trust. The tuple is provenance at the read, not trust at evidence commit, and cannot change AttemptKey, artifact/log validity, capture classification, or native Lakeflow/dbt outcomes.

P6 adds exactly one unpartitioned customer-security-owned managed Delta fence row at `Serializable` isolation. Its executable attended envelope creates the table with explicit `UTF8_BINARY`, sets isolation separately, adds eleven named checks with eleven separate `ALTER TABLE` statements, verifies every definition/property/partial state, and inserts the singleton last; inline `CREATE TABLE` checks are not supported and are prohibited. A conditional same-row compare-and-swap from `OPEN` to `ACTION_CLAIMED`, gated by one exact fresh `CONTROLLED_ACTIONS` summary, is the authorization linearization point. The action-ledger write and Jobs Run Now happen afterward; one deterministic 64-hex idempotency token is reused after response loss, and no workload parameter is sent. Get Run prefers current status/termination details and uses deprecated state only as an absent-current fallback; legacy `INTERNAL_ERROR` is a known terminal failure, while conflict or unknown evidence holds closed. A drain compare-and-swaps `OPEN` to `CLOSED` or a previously claimed action to `DRAINING`. The drain commit prevents later dbtobsb admissions, but the earlier action may still dispatch or run; it must be reconciled, cancelled or allowed to finish, and proven terminal with final run/task/compute inventory before `CLOSED` authorizes any trust/App/binding/refresh/upgrade/rollback/stop/uninstall mutation. Lease expiry allows only reconciled takeover. Acceptance never reopens automatically; the verifier explicitly moves the matching closed drain to `OPEN` against the new accepted snapshot. Uninstall ends in irreversible `RETIRED`.

This fence uses no multi-table transaction and is not atomic with Jobs. It coordinates honest product writers only; it does not block customer schedules/direct callers or resist App code/managers, the verifier, UC owners, or administrators. Those principals are explicit roots because the App holds table-wide fence `MODIFY` and approved-Job Run Now authority. Customers requiring protection from compromised App code must keep P6 disabled until a separately governed fixed dispatcher removes that direct authority.

The fresh-install App lifecycle uses three sequentially generated Direct plans, not plans captured from the same state. The first creates a stopped zero-resource App shell with no deployment or user `CAN_USE`. The second, generated from fresh shell state, adds the exact five read-only resources and their deploy-time `config.env[].value_from` mappings while permissions remain empty. Direct App Update applies the resources but cannot retrofit deploy-only config. The wrapper then requires an empty deployment set and invokes `bundle run <app-key>` exactly once against that current bound-unshared overlay. The runner starts compute before deploying and may reissue its deployment POST after an error, so the wrapper never retries, discloses the bounded read-only bound interval and named trusted roots, stops on every exit, and reconciles the fully paginated after-set. Only exactly one new terminal successful matching `SNAPSHOT` and no pending deployment may proceed; the user never chooses or enters an ID. An existing App with any deployment is not a fresh shell and must block or use a separately qualified upgrade path.

Only after deployment reconciliation and a proved `STOPPED` state does the wrapper calculate the executable user-access plan from fresh Direct and remote App state. A separate approval verifies new lineage/serial; unchanged source, build, App code, artifact, command, configuration, resource bindings, and reconciled deployment ID; `lifecycle.started:false`; and a diff limited to the approved group `CAN_USE`. Drift or an interrupted/partial Apply requires a fresh plan. The verifier then completes the pre-start GA observation and native roster review and appends only a non-authoritative candidate. `databricks apps start <app-name>` starts that same deployment without deploying code. The running App remains setup-only until a distinct post-start observation proves `ACTIVE`, the same deployment ID and stable graph, zero pending deployments, and an eligible original roster anchor, and one final `SNAPSHOT_ACCEPTED` row commits. No-argument or bare `apps deploy` is prohibited throughout the wrapper; `bundle run` and `lifecycle.started:true` are prohibited after the one bound deployment invocation. The App exposes the three evidence times and expiry without claiming continuous integrity. The bound deployment interval is fail-closed but not zero-authority: the App principal/code, deployment principal, App managers, workspace administrators, and Databricks control plane are explicit roots.

Because an `ALL_DONE` task is not guaranteed to run after whole-run cancellation or Job-level timeout, the same idempotent collector also has a bounded scheduled reconciliation mode and an operator-triggered mode. The initial recovery target is one 15-minute interval plus five minutes of scheduling delay. Reconciliation does not keep compute running between invocations, and test schedules start paused.

The required MVP App is read-only. Optional P6 controlled actions are enabled only through a separate reviewed runtime plan, secret, nine-table attended fixed privilege/data envelope including the closed fence, pending attestation plus composite seal, and runtime-trust upgrade; until then the App has no action identity key, action-role resources, fence DML, or `CAN_MANAGE_RUN` grant on observed Jobs. Even after acceptance, the verifier must explicitly reopen the fence.

Optional system-table enrichment is a separately installed capability and is `DISABLED` in the base product. When approved, its attended fixed migration extension creates one minimal product-owned `(workspace_id, job_id)` scope table, `lakeflow_job_run_snapshot`, `lakeflow_dbt_task_run_snapshot`, and `dbt_job_cost_snapshot` plus sanitized views. A separate paused fixed-code Job and dedicated service principal read only accepted `system.lakeflow` and `system.billing` schemas, join immediately to that scope table, allowlist columns, and write only the three snapshot tables. The attended UC operator is the scope table's sole writer through an approved idempotent fixed DML-only scope-update envelope. Lakeflow access exposes account workspaces in the current cloud region and billing access is account-global; this residual scope is explicit even though persisted rows are restricted to the current workspace and onboarded Jobs. The enrichment Job has no DDL, and the App never receives `system` access.

The GA-only controlled-action design uses the App principal for the platform call and a separate deny-by-default FastAPI policy for the human decision. `X-Forwarded-User` is an IdP-provided request value that may be a username or email, not a documented opaque Databricks principal ID. The App keeps it request-local and derives an installation-scoped HMAC fingerprint using a customer-controlled key in a dedicated Databricks secret scope. That fingerprint maps to a random `actor_id` and identity epoch; roles and production separation of duties use only the `actor_id`. Every browser profile also needs an independently approved random credential. The raw header, HMAC input/key, and browser credential are never stored or displayed.

An enrollment request is a short-lived single-use nonsecret locator, not a bearer authorization secret. A named human action-role administrator verifies the accountable person through an approved out-of-band process, reviews the immutable role/Job/environment/expiry summary and restricted governance reference, then invokes an unscheduled fixed-code role-administration Job. That Job runs as a separate service principal with narrow table and warehouse grants; the App cannot run the Job or modify its own role bindings. FastAPI rechecks actor, browser, scope, expiry, revision, revocation, and distinct production approver at prepare, approve, and run. Pseudonymous identity, device, governance-reference, and action records remain Personal Data. Same-value account recreation cannot be detected from the GA forwarded header alone, so the regulated path also requires unique customer identities, bounded binding/device expiry, explicit offboarding, and identity-epoch revocation. Preview App user authorization and `system.access.audit` enrichment remain optional.

The initial App implementation will evaluate the official React plus FastAPI template. FastAPI owns business and policy logic; the UI must not contain enforcement rules that can be bypassed through the API.

## Why this starting point

- It keeps data, identity, compute, audit, and retention in the customer workspace.
- Bundles provide source-controlled, reviewable, repeatable deployment for the App and supporting resources.
- Databricks Apps have dedicated service principals and explicit resource bindings.
- A private deployment can validate the product before Marketplace provider onboarding is complete.
- The same App can later expose an optional MCP adapter or attach a read-only Genie Agent without making AI part of the trusted capture path.

## Consequences

- Installation is more than `databricks bundle deploy`. The wrapper must verify the customer-owned typed schema, owners, groups, dedicated warehouse, and trusted roots; execute and recover the fixed data envelope; compute the composite seal; create the trust ledger/view; then execute shell-plan/apply, bound-unshared-plan/apply, empty-baseline/bound-deploy/reconcile/stop, user-access-plan/apply, and pre-start/roster/candidate/same-deployment-start/post-start/accept. It shows the App URL only after acceptance and exposes all three statement-time facts and validity clocks. P6 acceptance is followed by an explicit verifier-owned reopen.
- The regulated handoff needs one supported multi-user workstation, two distinct managed OS accounts, a system-owned local spool, and an operator-owned U2M profile in each account. The signed wrapper never transports or deletes another person's credential. Same-login separation and improvised cross-workstation transfer fail readiness; the exact supported OS/architecture remains a P3 release gate.
- The personal combined path needs one supported account/profile and explicit acceptance that it provides neither independent data observation nor independent SP-role review. It uses the same fixed data, pending attestation, composite state checks, runtime isolation, recovery, and lifecycle controls but no spool, capsule, account release, or actor return. Its provisional bootstrap target is 20 minutes of total active actor time including native roster review; automated polling, startup, and platform wait are measured separately.
- The App runtime is billed while running. Tests must use bounded windows and stop the App afterward.
- Customers must provide or explicitly approve the dedicated existing Unity Catalog schema, product-object and attestation-security owner groups, migration-operator group, object-scoped seal-verifier/trust-committer group, two eligible U2M actors for the regulated route, account-administrator authority on the verifier, exact parent-use handoffs, sanitized attestation/trust views, App-binding authority, named runtime trusted roots, and one auto-stopping installer-only SQL warehouse. Both operator groups need direct `CAN_MONITOR`; its full-query-text exposure is accepted explicitly. The workspace-admin verifier's effective warehouse `CAN_MANAGE` and persistent trust-ledger `SELECT`/`MODIFY` are accepted separately. A policy requiring a different third identity reviewer is unsupported in v1.
- Product state cannot live on App disk or in memory; App state is ephemeral.
- The App and collector have zero required public-internet runtime egress. Any dependency or tool download belongs to a separately disclosed build/deployment boundary with signed, checksum-pinned, locked, mirrored/offline or explicitly allowlisted inputs.
- The candidate tested toolchain is GA Databricks CLI `1.9.0`, pinned Direct engine, serverless environment version 5/Python `3.12.3`, `dbt-core==1.11.12`, and `dbt-databricks==1.12.2`. The customer-supplied installer-only serverless SQL warehouse must use the Current channel. Support is claimed only after fixtures and staging validation pass.
- CLI `1.9.0` is a checksum-pinned deployment-only dependency; the App and collector do not call it at runtime. Any CLI update requires fixture and staging requalification.
- Direct is the current GA/default engine for new Bundles and avoids the Terraform/HashiCorp dependency surface. The Bundle pins it explicitly so environment variables cannot change state engines. P3 must qualify its separate local/remote state, plan/diff, unsupported-field, and documented drift behavior across create, no-op, update, rollback, and destroy.
- The installer is a signed versioned `dbtobsb` wrapper. Every Direct plan binds CLI/profile/target/workspace/config/selection/source/build/remote state and lineage/serial. Shell, bound-unshared, and user-access plans contain only supported App/Job resources, ACLs, and verified existing-object bindings; Direct execution and hooks contain no migration Job/SP, schema resource/grants, SQL, UC DDL/DML, or ledger write. The shell is applied first without a deployment. The bound-unshared plan adds resources before the sole Bundle run carries deploy-only config into the only deployment. After immediate stop and reconciliation, the user-access plan adds only group `CAN_USE`. Explicit `databricks apps start <app-name>` later starts only that reconciled deployment. The bound deployment and safe-boot intervals are billed and bounded, and every failure defaults to stopped compute.
- Direct never manages the customer-owned schema's grants. CLI 1.9.0 treats remote direct principals omitted from desired grants as removals, and its saved plan carries plan-time remote state; using that as an authoritative schema list could revoke customer or App-generated access. The wrapper instead re-audits live direct and inherited privileges and changes only a named product-recorded principal/privilege pair through the approved envelope.
- The protected migration envelope shows the full fixed operation text and exact before/after owners, schemas, properties, grants, pending-attestation transition, and rollback class, but no component executes plan-supplied SQL. The signed wrapper re-renders every fixed statement in memory, re-snapshots state before each idempotent operation, and mutates only when the semantic digest matches. Every mutation has a prior Query History recovery marker. Multi-object DDL is journaled/idempotent rather than falsely described as transactional; drift blocks, failure enters cleanup first, and runtime remains unavailable until the remote attestation and fresh composite observation pass.
- A named customer product-object group owns retained product objects, and a separate named customer security group owns both control ledgers, sanitized views, and the P6 fence. Runtime principals retain no ownership by implication. The verifier's object-scoped `MANAGE`, persistent runtime-trust-ledger DML, and P6 fence DML are explicit roots, not tamper resistance. Persistent parent-use grants are explicit product records. Retain-uninstall drains and retires P6, removes App bindings first, preserves or transfers objects and trust/fence history, removes only exact product-recorded grants, and leaves the customer schema outside Bundle state. Delete-uninstall removes product objects/control ledgers/fence or the empty schema only through separately approved attended operations and verification.
- P6 action enablement separately plans and approves creation of a dedicated installation secret scope/key and the singleton fence before generating the App-binding Bundle plan. Secret values never enter argv, JSON plans, checkpoints, Job parameters, logs, or source control. Key rotation is an explicit migration or re-enrollment operation with a deadline and fail-closed compromise path. Every P6 lifecycle operation exposes the difference between admission blocked and action terminal, possible continuing cost, one recovery owner, and a separate reopen; it offers no force-open.
- System-table enrichment is optional, disabled by default, and isolated in a paused Job/service principal that materializes only three current-workspace/onboarded-Job snapshots from a product-owned pair allowlist. Lakeflow grants expose the account's workspaces in the current region and billing grants expose account-global data, so source access is an explicit residual-risk review item; the App, collector, UC migration path, and role Job receive no `system` access.
- Jobs output is not a receipt carrier in v1. Runs and output expire after 60 days, while the returning actor can read and verify the customer-owned sanitized Delta view directly.
- Job and code ownership are not hidden trust boundaries. Workspace admins retain general workspace-object management even when `RestrictWorkspaceAdmins` limits owner/run-as changes; Job owners/managers can alter tasks that still run as the configured service principal; and account/SP managers can delegate its use. A runtime-trust snapshot can detect GA-visible drift only when an attended observer refreshes it; the service-principal roster is administrator-attested, and a trusted administrator can bypass a code-level check. The App shows the observation age and never implies that changes after it are already detected. Customers requiring runtime evidence protected from these administrators or continuously machine-verified role custody receive `UNSUPPORTED_RUNTIME_CUSTODY_POLICY` in v1.
- The installer-only warehouse uses the lowest supported auto-stop and has a named manager for stop/escalation. Query History retention and full-text exposure are customer-controlled Databricks facts that dbtobsb neither copies nor claims to delete. Shared runtime SQL warehouse costs are labeled allocated or estimated, never exact per run. Pending and unavailable sources are visible states.
- Marketplace packaging remains a target, not an MVP dependency.
- Preview-only features remain optional. This includes App OpenTelemetry, Genie/MCP integrations, and Python Bundle configuration unless separately accepted.

## Rejected as the starting point

### External SaaS control plane

Rejected because it adds data egress, external telemetry, another identity boundary, and a larger regulated-industry review surface.

### Bundle and notebook only

Useful as an implementation mechanism, but insufficient as the target product because setup, health, migration, failure recovery, cost, and uninstall would remain difficult to understand.

### Privileged migration Job or migration service principal

Rejected because Job creator/owner and `CAN_MANAGE` authority can alter tasks, code, run-as, and permissions while the temporary data authority is active. Removing that authority also prevents the same deployment identity from later applying a Bundle that still manages the Job. A separate custody Bundle, workspace-admin ownership transfer, Service Principal User/Manager lifecycle, immutable artifact boundary, and explicit admin exception could make the pattern supportable, but it is larger and weaker than one attended U2M fixed-SQL migration path.

### UC human-owned migration Job

Rejected because it couples a production object to one person, lets that owner alter the privileged code path, complicates offboarding and upgrade, and still requires a workspace-admin owner transition. The UC operator instead uses their attended authority only for the bounded signed migration and durable seal.

### Marketplace-first implementation

Rejected as a dependency because third-party Marketplace Apps are new and provider submission details still require confirmation. The private App should establish product value and security evidence first.

### Genie or another LLM as the execution plane

Rejected because instructions and skills are guidance, not authorization or policy enforcement. Genie Agents are read-only SQL analysts, and Genie Code agentic capabilities may be unavailable in regulated workspaces.
