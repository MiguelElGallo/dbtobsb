# v0.4.0 supported-release contract

- Status: Release candidate pending local and live Azure qualification
- Distribution: Private Databricks App deployed through a Databricks Declarative Automation Bundle
- Marketplace: Out of scope
- Machine contract: `dbtobsb.support.v1` in `support-manifest-v1.json`
- Final contract digest: `00cd8d3633474ad8b5e8a9153aa55a44f02e00fc90ef61fbab7a8f302474a5e2`

## Supported outcome

One accountable administrator installs dbtobsb into one Azure Databricks workspace, onboards one supported dbt Core Job, captures deterministic dbt artifacts into customer-local Unity Catalog objects, and leaves the App stopped. An approved operator can run the dbt Job, and an approved viewer can explicitly start the read-only App and load the resulting observability data. The administrator can stop or uninstall the product without leaving product-started compute running.

The workspace must be a customer Azure Databricks workspace reached through its
canonical `adb-...azuredatabricks.net` per-workspace URL. Databricks Free Edition,
the retired Community Edition, AWS, and GCP are outside this contract.

The release is supported only when every condition below is true. Unsupported input fails before mutation and is not silently adapted.

The packaged `dbtobsb.support.v1` manifest is normative for exact machine values, generated commands, paths, objects, grants, App resources, lifecycle outcomes, and qualification cases. This page explains that contract; it does not create a second configurable source of truth.

## Supported people and authority

| Person or identity | May be different after installation? | Supported responsibility and authority |
|---|---|---|
| Combined administrator | No, exactly one human performs installation | Uses one managed macOS arm64 account and one named OAuth U2M profile. The same human selects, previews, approves, executes, and verifies production bootstrap. This person is an Azure Databricks account and workspace administrator, is the exact existing evidence-schema owner, and retains schema/object-owner authority. |
| dbt Job operator | Yes | May be a different approved user or group. Starts the observed Job through its normal Job permission. The Job runs as the dedicated observed service principal, not as the operator. |
| App viewer | Yes | May be a different approved group member with `CAN_USE` on the App. Evidence queries run as the App service principal with the fixed read-only bindings. |
| Observed service principal | Required and distinct | Runs the sealed product-owned dbt runner. It can write only the per-attempt staging Volume; it cannot read or modify normalized evidence. |
| Collector service principal | Required and distinct | Runs the collector and reconciler with only the fixed evidence-table, raw-Volume, and read-only staging privileges listed below. |
| App service principal | Created for the App | Receives only `CAN_USE` on the bound SQL warehouse and `SELECT` plus required parent usage on the three health views. |

`COMBINED_ROLE` provides no independent prevention, observation, approval, or service-principal-role review. The same administrator can install code, mutate the customer schema, change permissions, and verify the result. Catalog/metastore administrators, workspace administrators, the Job manager group, and App managers are also explicit trusted roots within their Databricks authority. dbtobsb detects neither all later administrator changes nor compromise of those trusted roots.

`SEPARATED_DUTIES` is not supported by v0.4.0. Any request for that or an unknown mode fails before Spark SQL, Bundle apply/run, Apps calls, Jobs calls, or native statement submission.

## One installation route

The canonical launcher is:

```console
dbtobsb bootstrap
```

It uses an explicit non-`DEFAULT` Databricks CLI OAuth U2M profile for the exact Azure host. Read-only preflight verifies the host, profile, current actor, existing schema, exact schema ownership, target identifiers, service principals, groups, warehouses, and prior product/compute state before showing one exact approval preview.

The selected Bundle target must be fresh or already use dbtobsb Direct state.
Databricks CLI 1.8.0's Terraform-to-Direct auto-migration is not an implicit
dbtobsb upgrade path; a target that resolves to Terraform state is outside this
release contract and must not be deployed by the launcher.

After approval, the launcher deploys and runs one temporary serverless Lakeflow Python-wheel bootstrap Job through the Bundle. The Job runs as the combined administrator who deployed it; `session_user()` must exactly equal the existing schema owner. The launcher waits for terminal readback and removes the temporary Job. Bundle Direct operations and Bundle hooks execute no SQL.

The native production-mutation registry remains foundation code and is not invoked by the v0.4.0 launcher. Bootstrap therefore has one execution plane, not a Spark path plus a hidden Statement Execution path.

Bootstrap uses serverless Job compute, not a SQL warehouse. The observed dbt task uses one customer-selected existing SQL warehouse. The App uses one customer-selected existing query warehouse; it may be the same warehouse or a different one. App compute is separate Databricks App serverless compute. The preview names all three roles and whether the two warehouse IDs are shared.

## Exact customer state

The empty evidence schema and observed-principal-owned dbt target schema may be
in different existing catalogs. Discovery, approval, the generated dbt profile,
and the runtime seal bind both fully qualified targets independently.

The installer does not create or delete the selected schema. Before product grants are added, bootstrap creates or verifies exactly these nine schema objects:

| Object | Kind | Purpose |
|---|---|---|
| `dbtobsb_object_manifest` | Managed Delta table | One sealed installation row |
| `dbt_artifact_registry` | Managed Delta table | Attempt, archive, capture, and collection state |
| `dbt_invocations` | Managed Delta table | Normalized dbt invocation outcomes |
| `dbt_node_results` | Managed Delta table | Normalized model, seed, snapshot, and test outcomes |
| `dbt_run_health` | View | Sanitized run-level App projection |
| `dbt_node_health` | View | Sanitized node-level App projection |
| `dbt_collection_health` | View | Sanitized collection/reconciliation projection |
| `dbtobsb_raw` | Managed Volume | Exact-byte custody for retrieved archives |
| `dbtobsb_stage` | Managed Volume | Per-attempt handoff of allowlisted dbt artifacts from the observed runner to the collector |

No runtime-trust ledger, runtime-trust status view, controlled-action table, system-table enrichment object, or additional customer object is part of the v0.4.0 supported path. The object manifest, finalized deployment binding, exact runtime artifacts, and least-privilege readback are the installation integrity controls for this release. Advanced runtime-trust foundation code does not make a supported-release claim.

All nine objects are owned by the combined administrator because that person owns the selected schema. The installer adds exactly these product privileges:

| Principal | Securable | Privileges | Provisioner |
|---|---|---|---|
| Collector service principal | Evidence catalog and schema | `USE CATALOG`, `USE SCHEMA` | Attended installer |
| Collector service principal | `dbtobsb_object_manifest` | `SELECT` | Attended installer |
| Collector service principal | Each of `dbt_artifact_registry`, `dbt_invocations`, `dbt_node_results` | `SELECT`, `MODIFY` | Attended installer |
| Collector service principal | `dbtobsb_raw` | `READ VOLUME`, `WRITE VOLUME` | Attended installer |
| Collector service principal | `dbtobsb_stage` | `READ VOLUME` | Attended installer |
| Observed service principal | Evidence catalog and schema | `USE CATALOG`, `USE SCHEMA` | Attended installer |
| Observed service principal | dbt target catalog | `USE CATALOG` | Attended installer |
| Observed service principal | `dbtobsb_stage` | `READ VOLUME`, `WRITE VOLUME` | Attended installer |
| App service principal | Evidence catalog and schema | `USE CATALOG`, `USE SCHEMA` | Databricks App resource binding |
| App service principal | Each of `dbt_run_health`, `dbt_node_health`, `dbt_collection_health` | `SELECT` | Databricks App resource binding |
| App service principal | App query warehouse | `CAN_USE` | Databricks App resource binding |

The observed service principal receives no table, view, or raw-custody grant. The App receives no table `MODIFY`, Volume, Job, or raw-artifact access. Unrelated pre-existing or later customer grants are never silently removed.

Fresh bootstrap and an exact pre-binding rerun are idempotent. Once collector or App grants exist, resume uses the durable lifecycle stage and exact remote inventory; it does not replay the pre-binding bootstrap or reject the installation merely because its own approved grants now exist. Partial objects, foreign objects, changed definitions, changed ownership, unexpected product-principal grants, or an indeterminate mutation block progress and route to readback. An ambiguous mutation is never automatically resent.

## Exact dbt Core boundary

The full supported runtime tuple is:

| Distribution | Exact version |
|---|---|
| Python | Databricks serverless environment 5 / Python `3.12.3` |
| `dbt-core` | `1.11.12` |
| `dbt-databricks` | `1.12.2` |
| `dbt-adapters` | `1.24.5` |
| `dbt-common` | `1.37.5` |
| `dbt-protos` | `1.0.541` |
| `dbt-spark` | `1.10.3` |
| `databricks-sdk` | `0.117.0` |
| `databricks-sql-connector` | `4.3.0` |
| Databricks CLI on the installer workstation | `1.8.0` |

The release artifact includes the checksummed resolved environment and pins the
official Databricks CLI `v1.8.0` darwin-arm64 executable, release commit, Go module
sum, and executable SHA-256. Version drift requires a new support manifest,
product release, local qualification, and live Azure qualification; an installed
customer policy cannot bless it.

`generate_dbt_commands()` and `expected_dbt_output()` are normative. They generate optional ordinal `000-deps` followed by required ordinal `001-build`, exactly one named selector, the six-axis AttemptKey, and isolated artifact/log paths. The `run-dbt` wheel entry point maps that sealed attempt root to task-local storage, runs dbt without a shell, and uploads only `manifest.json`, `run_results.json`, and bounded structured logs to `dbtobsb_stage` through the Databricks Files API with exact-byte readback. Operators cannot supply commands, flags, paths, `vars`, target overrides, environment overrides, or command fragments.

The AttemptKey axes are workspace ID, Job ID, Job run ID, dbt task run ID, Job repair count, and task execution count. The accepted nested members are:

- optional `target/dbtobsb/attempts/{AttemptKey}/000-deps/logs/dbt.log`;
- `target/dbtobsb/attempts/{AttemptKey}/001-build/artifacts/manifest.json`;
- `target/dbtobsb/attempts/{AttemptKey}/001-build/artifacts/run_results.json`; and
- `target/dbtobsb/attempts/{AttemptKey}/001-build/logs/dbt.log`.

The machine contract freezes JSON artifact writing, JSON/info file logs, 20 MiB per log file, six files and 120 MiB total logs, 128 MiB per primary artifact, disabled anonymous usage and artifact-ingest upload, `--no-fail-fast`, eager indirect selection, and warning errors for empty selection/nothing-to-do. An empty run-results document is not a successful supported run.

Supported artifacts are manifest v12, run-results v6, and structured log v3 at their packaged schema digests. One exact reviewed dbt-databricks 1.12.2 manifest-v12 exception is accepted: `macros.macro.dbt.materialization_function_default.supported_languages` may equal exactly `['sql', 'python', 'javascript']` at the frozen schema path. No other schema error is accepted.

Every successfully retrieved archive is written as exact bytes to `dbtobsb_raw` and read back before parsing or normalized publication. Raw custody is mandatory, not an optional feature. If retrieval produces no archive, no raw bytes exist and that absence is recorded explicitly.

## Real customer-project onboarding

The supported onboarding policy is `dbtobsb.dbt-policy.v1`:

- The observed task is exactly the `dbtobsb-collector` Python wheel entry point `run-dbt`; native dbt-task artifact archiving is not assumed.
- The installer produces a Bundle-relative immutable project snapshot, sealed runtime policy, and source-controlled Job patch; it does not mutate the Job before owner approval.
- Task key is `dbt_build`, environment key is `dbt`, and `profiles_directory` exactly matches the generated `project_directory`.
- The snapshot contains the customer project plus a product-generated project-root `profiles.yml`; `dbt_project.yml`, `profiles.yml`, and `selectors.yml` are mandatory. The source project's scalar `profile` value must match `^[A-Za-z][A-Za-z0-9_-]{0,127}$`; the generated profile uses that exact name (`DBT_PROJECT_YML_PROFILE_SCALAR`) and cannot select another profile at run time.
- The only target is named `dbtobsb`. Its output has exactly the keys `type`, `method`, `host`, `http_path`, `token`, `catalog`, `schema`, and `threads`: type is `databricks`, method is `http`, host is the installer-rendered `INSTALLER_RENDERED_CANONICAL_WORKSPACE_SERVER_HOSTNAME`, token is `{{ env_var('DBT_ACCESS_TOKEN') }}`, and threads is `1`. The host source is the `AUTHENTICATED_NAMED_DATABRICKS_CLI_PROFILE`, canonicalized by `HTTPS_AZURE_WORKSPACE_URL_TO_LOWERCASE_DNS_NAME`; the operator never enters it. The installer also renders the reviewed literal catalog/schema and the existing SQL warehouse path (`INSTALLER_RESOLVED_EXISTING_SQL_WAREHOUSE_HTTP_PATH`). No secret value is written into the snapshot.
- The dbt task omits `warehouse_id`, `catalog`, and `schema` and sets `profiles_directory` to the same generated snapshot directory. This deliberately selects the product-generated custom profile rather than Databricks' automatically generated warehouse profile. Databricks injects only the `DBT_ACCESS_TOKEN` used by this profile for the configured Job run-as principal; the operator supplies none of the connection or generated-command fields: profile name, target, token, host, HTTP path, warehouse ID, catalog, schema, flags, and paths are all derived or rendered by the product.
- Generated `logs/` and `target/` roots are excluded; every other approved source file has a normalized relative path and SHA-256 digest in a closed source manifest.
- Exactly one version-controlled selector matching `^[A-Za-z][A-Za-z0-9_-]{0,63}$` is installed.
- A project with no dbt dependencies omits ordinal `000-deps`. A project using `packages.yml` or `dependencies.yml` must commit a matching `package-lock.yml`; only then does the generated policy enable the fixed `dbt deps` command.
- The installed policy contains the support-contract digest, source-file digest map, source digest, selector, dependency decision/lock digest, task/environment keys, and canonical runtime-policy digest. Commands and paths are derived, never accepted from the caller.
- The generated review shows the dbt command/selector, run-as service principal, warehouse/catalog/schema target, collector `ALL_DONE` edge, exact source changes, and exact rollback to the previous Job definition.

Changing customer source, selector, or dependency files requires a fresh installation into an empty evidence schema. v0.4.0 does not update the sealed project policy in place. Runtime distributions, artifact schemas, log version, generated flags, ordinals, path templates, source type, and target-binding shape additionally require a new product release and qualification.

## Runtime collection and first App value

- The observed Job invokes the collector with `ALL_DONE`; Lakeflow/dbt failure and collection failure remain separate outcomes.
- The reconciler is deployed paused and performs only the bounded fixed scan when explicitly run or later enabled by an administrator.
- Runtime principals cannot switch from fixed collection DML to DDL through a Job parameter, dbt input, selector, artifact, or log.
- The App is deployed once with only the query warehouse and three read-only health
  views, stopped and reconciled, and then the exact approved group receives targeted
  `CAN_USE` through the Apps permission API while compute remains stopped. Successful
  installation leaves the App `STOPPED`.
- `dbtobsb start` requires an App-cost acknowledgement and starts the App without querying the warehouse.
- The landing page performs no evidence query. It explains that App compute is active, that explicit **Load observability** can auto-start the bound SQL warehouse, that the warehouse may incur separate cost, and that closing the browser stops neither resource.
- After **Load observability**, the App reads the three sanitized views. It distinguishes Lakeflow task outcome, artifact retrieval, artifact pair, dbt invocation, failed/skipped node status, and collection/reconciliation state, then provides one safe next action. Empty and query-failure states are actionable.
- The same explicit load renders two native, server-generated App charts: accepted node
  results with dbt status `error` or `fail` per run, and accepted model-result rows per run.
  “Total models” is a run-result count, not a claim about total project inventory. The
  charts add no external dashboard object, iframe, JavaScript dependency, view, table,
  grant, resource binding, or telemetry service. Their fixed aggregate query reads only
  `dbt_run_health` and `dbt_node_health` through the existing App bindings.
- No telemetry, artifact, log, prompt, support payload, identity, or Personal Data is sent to an external observability or AI platform.

## Lifecycle and cost contract

| Action | Command | Safe result | Approval and irreversible consequence |
|---|---|---|---|
| Fresh install or resume | `dbtobsb bootstrap` | Exact stage discovery; App `STOPPED`; runtime Jobs deployed; reconciler `PAUSED`; temporary bootstrap terminal and removed; warehouses unchanged | Exact change preview. Agent task authorization covers installer confirmations and retries within the same resources, mutations, workload, cost ceiling, and finish state; material expansion is a new task decision. |
| Start | `dbtobsb start` | App `ACTIVE`; SQL warehouse unchanged until explicit Load | App-compute cost acknowledgement |
| Stop — default safe action | `dbtobsb stop` | All product data, configuration, Jobs, bindings, and grants preserved; App `STOPPED`; reconciler `PAUSED`; runs terminal; product-started compute stopped or prior state restored | No destructive approval |
| Retain-uninstall | `dbtobsb uninstall --retain` | App, Jobs, product grants, bindings, and local installer state removed; all nine objects and raw archives preserved under the existing combined-administrator owner | Exact resource-removal and existing-owner retention approval |
| Delete-uninstall | `dbtobsb uninstall --delete` | App, Jobs, grants, bindings, and exactly the nine product objects removed; selected schema and unrelated state preserved | Separate destructive approval after retention, legal-hold, and required-export acknowledgement. Raw archives and normalized evidence are irreversibly deleted. |

The launcher stores a private resumable lifecycle stage and the exact selected product resources. Start, stop, and uninstall read current product state before mutation and do not report success until their terminal inventory matches the table. The launcher does not start or stop the selected SQL warehouse; dbt and explicit App queries use its existing Databricks lifecycle. Unrelated Apps, Jobs, warehouses, clusters, schemas, and objects are not changed.

Fresh install, resume, start, stop, retain-uninstall, and delete-uninstall are supported. In-place upgrade, rollback after installation, and adoption of an existing dbtobsb App deployment are not supported by v0.4.0.

## Regulated-use statement

dbtobsb v0.4.0 is designed for customer-local deployment without an external telemetry service, but it is not certified or attested against a regulatory framework. It makes no independent-control, administrator-resistance, continuous-monitoring, legal-retention, or compliance guarantee. A customer in a regulated industry must approve the combined-administrator topology, trusted roots, retention policy, App/warehouse cost behavior, and operational controls under its own governance before production use.

The `0.4.0` release candidate is not supported for production until its immutable
artifacts pass the local release suite and the complete live Azure matrix below. A
rebuilt or modified artifact set must repeat the applicable gates.

## Explicitly unsupported

- Databricks Marketplace, Databricks Free Edition, the retired Community Edition,
  AWS Databricks, or GCP Databricks;
- `SEPARATED_DUTIES`, independent-human-control, or independent-review claims;
- Windows, Linux, Intel macOS, or unmanaged installer hosts;
- creation or deletion of the customer-selected schema;
- quoted-only, punctuated, or Unicode catalog/schema names;
- versions, schemas, logs, flags, paths, or runtime packages outside `dbtobsb.support.v1`;
- Git-sourced dbt tasks, arbitrary dbt commands, multiple selectors, caller paths, `vars`, flags, or environment overrides;
- dependencies without a committed matching `package-lock.yml`;
- runtime-trust ledger/status, controlled actions, system-table enrichment, Genie, external AI, or external telemetry;
- automatic replay of an indeterminate mutation;
- in-place upgrade, project re-onboarding, rollback, or existing-deployment adoption; and
- production use of rebuilt or modified artifacts before the local and live Azure gates pass.

## Release gates

### Local release candidate

One immutable release source and support manifest must prove:

1. reproducible checksummed packages, native helper, resolved Python environment, support manifest, SBOM, and Bundle;
2. contract parity across the manifest, command generator, parser, collector, App bindings, lifecycle table, and this page;
3. deterministic non-demo onboarding and byte-identical patch/policy regeneration;
4. fixtures for success, dbt node failure, early failure, partial artifact, retry, repair, missed collection, identical replay, conflicting replay, reconciliation, empty App, App query failure, and exact trend aggregation;
5. install/resume/start/stop/retain/delete interruption injection with no duplicate mutation or false success;
6. App fixtures proving zero evidence queries before Load, exact outcome/next-action labels afterward, and accessible failures/model-result charts with a tabular fallback; and
7. complete quality, security, documentation, publication-safety, and domain-review gates with no unresolved critical/high finding.

### Live Azure proof

A behavior-changing release must prove the complete matrix below from the same immutable source and support manifest in the paid test workspace. The functional run matrix uses one installation seal. Retain- and delete-uninstall are terminal alternatives, so their rehearsals may use fresh installations whose resource-bound installation and artifact seals differ while the source-contract and runtime-policy digests remain identical.

The `0.4.0` App behavior and support manifest changed, so the complete live matrix
must run from the same immutable release source. Prior `0.3.0` evidence is historical
context only and cannot qualify this release.

1. one clean combined-role installation using the released launcher and temporary bootstrap Job;
2. exact actor, schema owner, objects, grants, App bindings, Jobs, warehouses, and released artifact digests;
3. one non-demo project onboarded without entered SQL, IDs, flags, paths, `vars`, or environment overrides;
4. two `COMPLETE`/`PAIR_VALID` runs of the same deterministic seed revision with distinct attempt/archive/invocation histories;
5. one dbt failure or partial-artifact path without relabeling the native dbt outcome;
6. one deliberately missed collection recovered by the bounded reconciler without rewriting the original attempt;
7. SQL assertions over exact run/node/collection rows, exact per-run trend aggregates,
   and matching App/API results;
8. start, stop, retain-uninstall, and delete-uninstall receipts matching their previews; and
9. final paginated inventory proving the App stopped or removed, reconciler paused or removed, runs/statements terminal, temporary bootstrap removed, and zero product-started compute left running.
