# Wire a supported dbt Job

Use this guide to onboard one dbt Core project into the sealed dbtobsb observed Job. The supported release does not scan or rewrite arbitrary Jobs and does not depend on Databricks returning a native dbt artifact link.

## Generate the immutable project and Job patch

Run the onboarding command from the Bundle root. Supply the target document on standard input so connection values are not copied into shell arguments:

```console
printf '%s' '<reviewed-target-json>' \
  | uv run --project installer dbtobsb-onboard-dbt-project \
      --source-project '<dbt-project-directory>' \
      --bundle-root .
```

Review the generated project snapshot, `dbt-policy-v1.json`, and `.dbtobsb-observed.generated.yml`. The source contract hashes every project file. A later file, selector, profile, command, target, or dependency change requires a new onboarding result.

## Understand the observed task

The generated task is a Python-wheel task, not a native `dbt_task`:

```yaml
python_wheel_task:
  package_name: dbtobsb-collector
  entry_point: run-dbt
  parameters:
    - --workspace_id
    - "{{workspace.id}}"
    # The remaining correlation values and immutable project/policy paths
    # are generated by the installer.
```

The Bundle uses serverless environment client `5` and pins the complete qualified graph, including dbt Core `1.11.12`, dbt-databricks `1.12.2`, and Databricks SDK `0.117.0`.

At runtime the runner:

1. verifies the packaged policy, workspace, project path, and six-axis attempt identity;
2. obtains the task identity's short-lived OAuth bearer credential in memory;
3. runs the sealed dbt commands without a shell in task-local storage;
4. uploads only `manifest.json`, `run_results.json`, and bounded JSON dbt logs to the fixed per-attempt path in `dbtobsb_stage`; and
5. downloads every uploaded file again and verifies its size and SHA-256.

The runner returns dbt's exit code. A failed dbt invocation can therefore leave useful partial evidence for collection.

## Keep the collector edge

The generated observed Job invokes the fixed collector Job after the runner:

```yaml
- task_key: collect_dbt_evidence
  depends_on:
    - task_key: dbt_build
  run_if: ALL_DONE
  run_job_task:
    job_id: ${resources.jobs.dbtobsb_collector.id}
    job_parameters:
      workspace_id: "{{workspace.id}}"
      observed_job_id: "{{job.id}}"
      observed_job_run_id: "{{job.run_id}}"
      dbt_task_run_id: "{{tasks.dbt_build.run_id}}"
      observed_task_key: dbt_build
      repair_count: "{{job.repair_count}}"
      execution_count: "{{tasks.dbt_build.execution_count}}"
```

`ALL_DONE` is essential. The collector verifies the child invocation, parent Job, installed Job graph, run identity, project bytes, terminal task state, and attempt coordinates before opening Spark or reading Unity Catalog.

Do not ask an operator to copy a Job ID, run ID, task-run ID, Volume path, selector, SQL statement, or credential. These are generated and sealed installation values.

## Understand what is captured

The observed principal can read and write only its staging Volume. The collector can read staging, write exact-byte custody to `dbtobsb_raw`, and publish the normalized tables. It reconstructs a deterministic bounded archive containing only:

```text
target/dbtobsb/attempts/{AttemptKey}/001-build/artifacts/manifest.json
target/dbtobsb/attempts/{AttemptKey}/001-build/artifacts/run_results.json
target/dbtobsb/attempts/{AttemptKey}/001-build/logs/dbt.log
```

When `dbt deps` is part of the sealed policy, its bounded structured log is included under ordinal `000-deps`. Compiled SQL, `catalog.json`, environment dumps, and arbitrary files are never staged by the runner.

## Verify the result

Run the observed Job and query the curated views:

```sql
SELECT capture_state, pair_state, lakeflow_result_state, issue_code
FROM `<catalog>`.`<schema>`.`dbt_run_health`
ORDER BY task_start_time DESC;
```

For a complete pair, `capture_state` is `COMPLETE` and `pair_state` is `PAIR_VALID`. A failed dbt build can still have those evidence states; inspect `lakeflow_result_state` and node statuses separately.

If capture is not complete, follow [Recover a non-complete capture](recover-capture-states.md).
