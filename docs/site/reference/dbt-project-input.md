# dbt project input

This page defines the dbt Core project that `dbtobsb bootstrap` accepts in version
`0.3.0`. For the step-by-step path, use
[Prepare a dbt project](../how-to-guides/add-a-dbt-project.md).

## Source and installed files

| Path | Source project | Installed snapshot |
| --- | --- | --- |
| `dbt_project.yml` | Required | Copied and hashed |
| `selectors.yml` | Required | Copied and hashed |
| `profiles.yml` | Ignored if present | Generated by dbtobsb |
| `models/`, `seeds/`, `tests/`, and other ordinary project files | Optional | Copied and hashed |
| `logs/` and `target/` | Ignored | Generated separately for each run |

The source project must be an isolated child directory below the dbtobsb repository
root. It cannot be the repository root itself. Files must be regular files with
ordinary relative paths. Hidden files, hidden directories, and symbolic links are
rejected. One file may be at most 128 MiB, and the accepted source files together
may be at most 512 MiB.

## `dbt_project.yml`

The document must be valid YAML with no duplicate keys. Its scalar `profile` value
must match:

```text
^[A-Za-z][A-Za-z0-9_-]{0,127}$
```

The top-level `flags`, `log-path`, `packages-install-path`, and `target-path` keys
are not accepted. dbtobsb sets those values in the fixed runtime command so logs
and result files use the installed evidence path.

See the general dbt
[`dbt_project.yml` reference](https://docs.getdbt.com/reference/dbt_project.yml).

## `selectors.yml`

The document must contain only one top-level key named `selectors`. Its value must
be a list containing exactly one named selector. The name must match:

```text
^[A-Za-z][A-Za-z0-9_-]{0,63}$
```

Jinja is not accepted in this file. dbt Core must be able to parse the selector,
and the selector must produce at least one executable result for the observed
`dbt build`. See the general dbt
[YAML selector reference](https://docs.getdbt.com/reference/node-selection/yaml-selectors).

## Dependencies

A project without packages needs no dependency files. A project with packages must
contain exactly one of `packages.yml` or `dependencies.yml`, plus its matching
committed `package-lock.yml`. Jinja is not accepted in these files. The installer
verifies the lock and rejects missing, unexpected, ambiguous, invalid, or mismatched
dependency files.

## Generated profile

dbtobsb uses the `profile` name from `dbt_project.yml` and creates one target named
`dbtobsb`. It fills the profile from the approved Azure Databricks workspace,
warehouse, catalog, target schema, and observed Job identity. The token value is an
environment-variable reference; no token is stored in the installed snapshot.

Changing the project, selector, profile name, dependency graph, or target requires
a fresh installation and a new empty evidence schema.
