## Final smoke-wrapper assessment

### Operator setup and identity safety — Pass

- Required profile, workspace host, and signed-in user variables fail immediately with instructions naming the expected value.
- Invalid stop-verification retry values and missing `databricks`, `jq`, or `curl` commands produce short, specific errors before mutation.
- The exact supported Databricks CLI version is checked before deployment and matches the bundle constraint.
- The named profile must resolve to OAuth U2M for the exact expected host and user before deploy or start.
- Ambient token/client-secret authentication and shell tracing are rejected with the reason stated in plain language.

### Credential and output safety — Pass

- The OAuth token is sent to `curl` through standard input, not a process argument, and is unset immediately afterward.
- `curl --disable --proto '=https' --tlsv1.2 --config -` prevents user configuration and non-HTTPS protocol fallback from changing the bounded request.
- Tests prove that the token is absent from `curl` arguments, standard output, and standard error.
- App responses and logs contain only allowlisted smoke metadata. They disclose no token, host, user, workspace identifier, filesystem path, data row, dbt artifact, or exception text.
- Recovery commands use Bash `%q` escaping, so an unusual profile name remains a single safe shell argument rather than becoming executable syntax.

### Cost and recovery behavior — Pass

- The App declaration is stopped by default and has no declared resource bindings; the copy says an explicit start uses billable App compute.
- Before starting, the wrapper verifies both remote `STOPPED` state and absence of live resource bindings.
- After starting, it verifies `ACTIVE`, the expected App URL shape, and again verifies there are no live resource bindings.
- The cleanup trap is installed before deployment and always attempts the idempotent stop once deployment might have created or drifted the App.
- Cleanup covers normal exit, command failure, interrupt, and termination. State reads are retried after stop so a transient read failure does not create false uncertainty.
- A verified stop produces the explicit final message `STOP VERIFIED: App compute state is STOPPED.`
- Failed or unverified cleanup is prominent, warns that cost may continue, prints exact manual recovery commands, and cannot produce a successful exit.
- Executable tests prove cleanup after success, cleanup after health failure, recovery from the first failed cleanup-state read, and stop-without-start when an unexpected live resource binding is found.

The copy accurately separates “smoke verified” from “stop verified”; it never equates process liveness with dbt capture, storage, authorization, or product readiness.

## Accessibility-relevant disposition

There is no interactive HTML interface in this P0 shell, so a rendered WCAG review is not applicable to the shipped surface. The API uses small structured JSON responses. The wrapper uses linear plain text without prompts, animation, cursor control, or color-dependent meaning. Failure messages begin with a recognizable condition such as `STOP FAILED` or `STOP UNVERIFIED`, followed by consequence and recovery commands, making them understandable with screen readers and in copied logs.

If an HTML UI or interactive API documentation is introduced later, it requires a separate keyboard, focus, headings/labels, zoom/reflow, contrast, and external-asset review.

## Coverage note

The subprocess suite verifies the exact success confirmation and multiple cleanup/error paths. The manual recovery function was additionally inspected and its `%q`-rendered commands were reproduced locally. A future test that forces the `apps stop` command itself to fail would strengthen regression coverage for those two error lines, but the current implementation is direct, shell-checked, and correct; this is not an open usability finding.
