# P0 Databricks App smoke usability and accessibility review

**Verdict: CHANGES_REQUIRED**

The implementation is a good bounded HTTP smoke shell: it is stopped by default, declares no product resource bindings, returns small allowlisted responses, uses `/api/health`, and does not expose identifiers, configuration, paths, or exception text. It is not yet ready for a regulated-user smoke deployment because its undocumented HTML documentation routes load public third-party assets, its liveness response can be mistaken for product readiness, and the advertised OpenAPI route is outside the Databricks App `/api/` route convention used for token-authenticated API access.

## Review method and evidence

The review inspected the response contracts, generated OpenAPI document, default FastAPI documentation routes, bundle lifecycle declaration, dependency lock, and tests. Local checks on the frozen input passed:

```text
uv run --locked --extra dev pytest
3 passed in 0.10s

uv run --locked --extra dev ruff check dbtobsb_app tests
All checks passed!

uv run --locked --extra dev ruff format --check dbtobsb_app tests
3 files already formatted

uv run --locked --extra dev ty check dbtobsb_app tests
All checks passed!
```

Static bundle inspection confirmed `lifecycle.started: false`, no App `resources` bindings, and a default `smoke` target. This review deliberately made no cloud or authentication calls.

The following primary references informed the findings:

- [Connect to a Databricks App locally](https://learn.microsoft.com/en-us/azure/databricks/dev-tools/databricks-apps/connect-local) states that token-authenticated Databricks App API endpoints use `/api/` routes.
- [Databricks Apps key concepts: app state](https://learn.microsoft.com/en-us/azure/databricks/dev-tools/databricks-apps/key-concepts#app-state) distinguishes running and stopped state and explains App compute charging behavior.
- [FastAPI metadata and documentation URLs](https://fastapi.tiangolo.com/tutorial/metadata/) documents explicit OpenAPI, Swagger UI, ReDoc, tag, and operation metadata configuration.
- [FastAPI self-hosted documentation assets](https://fastapi.tiangolo.com/how-to/custom-docs-ui-assets/) documents self-hosting the JavaScript and CSS needed for documentation in environments without public internet access.
- [WCAG 2.2](https://www.w3.org/TR/WCAG22/) and [Understanding descriptive headings and labels](https://www.w3.org/WAI/WCAG22/Understanding/headings-and-labels.html) provide the accessibility baseline if an interactive documentation page is shipped.

## Findings

### UX-SMOKE-001 — High — Default documentation pages silently require public third-party assets

`FastAPI(...)` leaves both interactive documentation defaults enabled. Local requests to `/docs` and `/redoc` return `200`, but the resulting HTML references assets from JSDelivr, Google Fonts, and `fastapi.tiangolo.com`.

This creates three user-facing problems in the stated regulated, Databricks-only environment:

- the pages are incomplete or unusable when public internet access is blocked;
- opening them creates browser requests to services outside the customer's Databricks boundary and may disclose request-context metadata according to browser and proxy policy; and
- they add complete interactive HTML surfaces that have not been included in accessibility, keyboard, zoom/reflow, or security testing.

The routes are especially easy to miss because `/` links only to the raw OpenAPI JSON, while users familiar with FastAPI may discover `/docs` themselves.

**Required change:** make the decision explicit. For this P0 smoke, set `docs_url=None` and `redoc_url=None`, retain a customer-local raw OpenAPI document, and test that both routes return `404`. If interactive documentation is intentionally shipped later, use checksum-pinned self-hosted assets with no public URL dependencies, make the page discoverable, and include the rendered page in accessibility and security review.

### UX-SMOKE-002 — Medium — `ok` communicates process liveness but can be read as product readiness

The endpoint implementation and docstring correctly test only whether the process can serve a request. The response itself contains only `status: "ok"`, service, and version. Neither `/` nor `/api/health` tells an operator that this is an unbound P0 smoke, that data/job/secret/warehouse/model-serving readiness is not evaluated, or that starting the stopped App uses App compute.

This matters because the user-visible word `ok` can reasonably be interpreted as “the observability product is installed and working,” even though the shell has no product bindings. The stopped-by-default property is understandable to a bundle reviewer, but it is neither expressed in the running service contract nor protected by a test.

**Required change:** expose an explicit, machine-readable distinction such as `stage: "p0_smoke"`, `health_kind: "process_liveness"`, and `readiness: "not_evaluated"`. The root index should state the available capability and the absence of product bindings in concise, non-alarming language. Update the bundle description to say that it is stopped by default, that starting it uses App compute, and that it has no product resource bindings. Add a configuration contract test that parses `databricks.yml` and proves `lifecycle.started` is false and that no App resource bindings are present.

Do not report the static lifecycle as “stopped” from a running process; a process serving this response has necessarily been started. Describe the deployment default and the current liveness semantics separately.

### UX-SMOKE-003 — Medium — OpenAPI discovery and generated client ergonomics are incomplete

The root advertises `/openapi.json`, while the callable health route is `/api/health`. Databricks documents `/api/` as the route space for token-authenticated App APIs, so a token-based consumer may be able to check health but not retrieve the contract through the same supported path.

The generated contract also exposes implementation-derived operation IDs (`service_index__get` and `health_api_health_get`), generic operation/response labels, tag names without descriptions, and fields without descriptions or examples. Because the raw schema is the only linked documentation surface, these details materially affect discoverability, screen-reader context in any later rendered documentation, and generated-client stability.

**Required change:** configure the schema at `/api/openapi.json`, update the service index, and test that exact route. Give each operation a stable explicit `operation_id`, descriptive summary, and truthful response description. Add concise tag metadata and field descriptions/examples that identify P0 smoke mode and process liveness without implying data, job, or product readiness.

## Positive controls to preserve

- `/api/health` follows the documented Databricks App API route convention.
- Both successful responses are small, predictable, and free of workspace, host, identity, profile, file-path, secret, and exception details.
- The health log line is allowlisted and non-sensitive.
- Tests assert exact successful response bodies instead of accepting arbitrary fields.
- The bundle declares `lifecycle.started: false` and contains no product resource bindings.
- The dependency resolution is locked, and the test, lint, formatting, and type gates pass.

## Required acceptance evidence

The following evidence is required before changing this verdict:

1. The existing locked `pytest`, Ruff check, Ruff format check, and `ty` commands pass.
2. Contract tests prove the exact smoke/liveness semantics and ensure no response or OpenAPI description claims job, data, or product readiness.
3. A bundle configuration test proves `lifecycle.started: false`, the default `smoke` target, and the absence of product resource bindings.
4. `/docs` and `/redoc` return `404`; alternatively, an approved self-hosted documentation implementation passes an asset scan showing no public URLs and a rendered accessibility review covering keyboard operation, focus order/visibility, headings and labels, 200% zoom, and reflow.
5. The root points to `/api/openapi.json`, and a contract snapshot asserts stable operation IDs, route summaries, response descriptions, tag descriptions, and schema field descriptions/examples.
6. A bounded Databricks smoke run proves that a consumer can fetch both `/api/health` and `/api/openapi.json` through the intended authentication path. That run must be started only by explicit action and end with inventory evidence that the App is stopped and no product bindings were created.
7. A served-HTML and network-request scan proves that the smoke surface makes no requests to public documentation asset hosts.

Once these changes and evidence are present, the usability/accessibility gate can be re-reviewed on a new frozen file hash.
