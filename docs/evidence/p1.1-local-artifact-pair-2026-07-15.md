# P1.1 local artifact-pair evidence — 2026-07-15

This record covers the offline P1.1 inspector only. It is not a Databricks runtime capture and does not qualify the candidate support row.

## Outcome

The inspector validated the three checked-in synthetic fixtures deterministically:

| Fixture | Pair result | Native dbt result | Intended lesson |
| --- | --- | --- | --- |
| `valid_success` | `PAIR_VALID` | `success=1` | Valid evidence and successful dbt outcome are separate facts. |
| `valid_dbt_failure` | `PAIR_VALID` | `error=1` | A valid evidence pair does not mean dbt succeeded. |
| `invalid_invocation_mismatch` | `PAIR_INVALID` | not interpreted | Files from different invocations cannot be trusted as one pair. |

Every fixture is generated by `capture/scripts/generate_artifact_pair_fixtures.py`, contains leak canaries, has a checksum record, and sets `runtime_evidence=false`. Run-results are synthetic. The generator consumes the checked-in reviewed synthetic projection `capture/tests/fixture_source/approved-manifest.json`; before copying any resource it requires that file's exact SHA-256, validates it against the pinned full schema, and verifies its expected resource and macro inventory. The quality gate regenerates every fixture in a temporary directory and requires a byte-identical comparison. An adversarial test changes origin metadata and injects unapproved free text; generation rejects it before creating an output directory. The projection was prepared from the separate local `dbt parse` compatibility exercise; neither that parse nor the sanitized fixtures prove Azure Databricks runtime behavior.
