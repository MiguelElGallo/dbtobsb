## What was deployed

The Databricks Bundle used a restricted `/Workspace/dbtobsb` root rather than `/Shared`. The observed and collector Jobs ran as distinct service principals. The finalized environment used Python environment client 5, dbt Core `1.11.12`, dbt-databricks `1.12.2`, and Databricks SDK `0.117.0`.

The two-phase artifact candidate was `f433e72b1499ab6df3f5f89d704c35d756541d8ddadfd9777b5f42fc80aaa3bf`. Candidate wheel paths were deployed and read back before promotion to the content-addressed final wheel set. The final content version was `6b52f8dd6e0f5ee90b8c2e294fd21bab0b23d0cd1c260968476f1130d8f2fa57`.

## Clean bootstrap

The evidence schema was recreated empty. A one-time serverless bootstrap ran as the combined administrator and emitted:

```json
{"event":"dbtobsb_bootstrap_verified","manifest_version":"dbtobsb.evidence.v1.0.0-rc.11","object_count":7}
```

The seven relational objects and both managed Volumes were present before runtime grants were applied. The observed identity received only parent usage plus read/write staging access. The collector received parent usage, fixed evidence-table DML, raw-Volume read/write, and staging read access.

## Cases still required before a final v0.3 tag

This run accepts the primary success and read-only App path. The machine support contract still requires live proof of a dbt failure or partial artifact, missed-collection reconciliation, retain-uninstall, and delete-uninstall. The attended run also used maintainer release tooling; the documented single-command installer launcher must be implemented and exercised before the contract can move from `CANDIDATE` to a final release.

Marketplace packaging, separated duties, compliance certification, and external telemetry remain out of scope.
