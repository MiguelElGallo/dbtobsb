# Repository working agreement

These rules apply to human and AI contributors.

## Repository install-and-run skill

When a user asks an agent to install dbtobsb, run the weather example or another
dbt project, make a first observed run, or prove that logs were captured, use
`.agents/skills/install-and-run-dbtobsb/SKILL.md`.

The skill must ask for resource, mutation, cost, project, warehouse-stop, and finish
choices before changing local or cloud state. It must pause again at the exact
installer preview before typing `APPROVE`. Do not replace the supported launcher
with direct dbt commands, caller-selected Job IDs, selectors, SQL, paths, flags, or
compute overrides.

The supported platform is a customer Azure Databricks workspace with a canonical
`adb-...azuredatabricks.net` per-workspace URL. Before mutation, reject AWS,
GCP, Databricks Free Edition, and the retired Community Edition. Databricks does
not currently call the personal-use product “Personal Edition”; personal-use
signup means Free Edition and is unsupported by dbtobsb.

For the v0.4.0 supported release, the machine-readable support manifest and
`docs/releases/v0.4.0-support-contract.md` are authoritative about included
components. The detailed separated-duty, migration-ledger, runtime-trust,
controlled-action, enrichment, upgrade, and rollback rules below apply only
when a later support manifest includes those components. They must not be
silently presented as v0.4.0 behavior or made a hidden dependency of its base
collector and read-only App.

## Product boundaries

- Keep all required product data and compute inside the customer's Databricks workspace.
- Do not add a required external telemetry, control-plane, licensing, or analytics service.
- Do not make Genie Code, Genie Agents, an MCP server, or another LLM feature a prerequisite for installation, capture, or operation.
- Do not let users or models provide arbitrary dbt commands, selectors, flags, job IDs, Git refs, SQL, target paths, log paths, or compute configuration.
- Do not write an active `dbt.log` directly to a Unity Catalog Volume. Complete append-style files locally, then copy an allowlisted closed file when policy permits.
- Give the collector two explicit, mutually exclusive authority modes. `BOOTSTRAP_ALLOWED` may execute only versioned, fixed, idempotent product DDL against the dedicated configured schema during an attended install or a clearly labeled personal/test self-bootstrap. `RUNTIME_DML_ONLY` is the steady-state mode and may write only the enumerated product tables. Never accept user-, App-, AI-, Job-parameter-, or plan-supplied SQL in either mode. Production and regulated guidance must make the bootstrap authority, actor, duration, verification, and revocation or retained residual authority explicit. Do not claim platform-enforced zero DDL authority: table-level `MODIFY` retains residual schema-alter authority, so the collector principal, its code deployers, and Job managers are trusted roots.
- Treat the customer-owned dedicated catalog/schema, owner or data-admin group, retention policy, and App-binding authority as typed, verified installer inputs. Do not let the Bundle create or adopt the schema.
- Do not configure an authoritative Direct `grants` list on the customer-owned schema. Preserve unrelated customer and App-generated assignments; add or remove only an exact product-recorded principal/privilege pair through the approved targeted privilege plane.
- Do not claim that a Direct Bundle plan creates managed Delta tables/views. App `uc_securable` bindings may reference only independently verified existing objects.
- Keep every saved Direct plan to supported runtime Job/App resources, bindings, and their ACLs. Direct plan/apply and Bundle hooks execute no SQL and hold no Unity Catalog DDL/DML authority: they contain no migration Job, migration service principal, schema resource/grant, table/view DDL, ledger write, or hidden data mutation. The resources deployed by that graph later run under separate principals with only their enumerated runtime DML.
- Do not hide SQL or another remote mutation in a Bundle pre-deploy script. Resource, data, seal, and runtime-binding mutations require their documented plans and approvals.
- Use one protected deterministic migration envelope for fixed targeted parent-use operations, DDL, idempotent data changes, object ownership, non-App grants, verification, the exact temporary migration-ledger grant/revoke when needed, and a pre-revoke attestation. There is no privileged migration Job or migration run-as service principal.
- Re-render every approved signed statement from versioned code. Execute the complete fixed envelope only in memory through the Statement Execution API as the attended OAuth U2M Unity Catalog operator; never execute plan-, user-, App-, AI-, or Job-supplied SQL, and never put rendered SQL in argv, Job parameters, checkpoints, logs, screenshots, or ordinary receipts.
- Claim separated duties only for exactly two people on one supported customer-controlled workstation: an account-and-workspace-admin deployment/seal verifier and a different UC operator in distinct managed OS accounts. The verifier also performs the native service-principal-roster review; a third reviewer, non-account-admin verifier, shared OS account, separate workstations, improvised transfer, or copied profile/checkpoint/token is unsupported in v1. Record `data_mutation_observer_separation=true` and `independent_sp_role_reviewer=false`. A personal `COMBINED_ROLE` uses one accountable account-administrator actor/account/profile and records every independence flag false. Mode becomes immutable after the first mutation; an incomplete operation cannot change mode to bypass cleanup.
- Keep every actor's complete OAuth U2M profile and native-secure-store entry inside that actor's OS account. Use an explicit profile and canonical host for every CLI and SDK call, force secure storage, reject inherited credential-bearing environment values or plaintext fallback, and never create, overwrite, copy, log out, or delete an operator-owned profile.
- In separated mode, use only the system-owned, auto-discovered local handoff spool for cross-account coordination. Capsules are signed, expiring, replay-resistant, atomic, digest-bound, and nonsecret; they contain no token, raw identity, SQL, full checkpoint, config/checkpoint path, private key, or signed URL. A local capsule is only navigation state and never authorizes runtime deployment. Combined mode must not emit or require a capsule, OS-account-release proof, or deployer-return state; the same actor resumes cleanup from their own private checkpoint.
- Require a customer-supplied installer-only SQL warehouse. The approved migration-operator and seal-verifier groups receive direct `CAN_MONITOR`, whose full-query-history exposure is a reviewed security and Personal Data consequence; do not reuse a warehouse containing unrelated queries. A seal verifier who is also a workspace administrator has effective `CAN_MANAGE` on the warehouse regardless of that direct ACL and must be disclosed as able to stop, edit, delete, and change permissions. Use exactly one operation per Statement Execution request with `wait_timeout=50s`, `on_wait_timeout=CANCEL`, bounded client timeouts, and no automatic retry of an indeterminate mutating POST. Only the submitting actor fetches Statement Execution results. A different operator uses the GA Query History API/UI, never the Preview system table or query tags, and treats accepted cancellation as nonterminal until Query History and live Unity Catalog state agree.
- Before each mutating statement, execute and verify one read-only preparation marker on that warehouse. It contains a signed, closed cleanup locator for the installation/generation, operation UUID, envelope/statement digests, exact customer-managed migration-operator group, securable, privilege pair, and intended action. The marker deliberately contains security metadata visible to every warehouse monitor; it contains no individual username, email, token, or SQL supplied by a user. Do not submit the mutation unless the marker is uniquely recoverable through Query History.
- The UC operator owns fixed-SQL execution, idempotent recovery, exact migration-group grant revoke, data verification, and the `DATA_APPLIED_PENDING_REVOKE` attestation in one OS session. Never return normal control while data mutation, an indeterminate query, or revoke work remains. A different eligible recovery actor—another migration operator or the separated deployment/seal verifier using its disclosed object-scoped `MANAGE`—may locate/cancel through Query History, wait to the frozen deadline, reconstruct live state, and perform only exact signed cleanup. Ambiguous or nonterminal state escalates to the warehouse manager and blocks safely.
- Before any installer mutation or resume, audit Query History, running compute/pending App deployments, every product object and owner, direct/inherited grants, implied `ALL PRIVILEGES`, effective `MANAGE`, visibility completeness, trust-event generations, and every result page. Use GA `SHOW GROUPS WITH USER`/`WITH GROUP` only for named migration actors/groups needed to classify their paths. Treat every granted group as a governed trusted root; do not claim complete or continuous member snapshots. Do not start ordinary product work until every query is terminal, the exact temporary migration pair is absent, live authority is classified, and App/trust lifecycle is in the one allowed state. Preserve foreign grants. Report current DML, ownership/`MANAGE` self-grant capability, and named roots separately.
- Make one customer-security-owned Delta migration ledger and sanitized current-attestation view the durable data-phase authority. The attended UC operator writes one closed, digest-bound, idempotent `DATA_APPLIED_PENDING_REVOKE` row containing completed prior Statement Execution IDs; the row cannot include its own statement ID and never claims to observe the later revoke. A matching retry is a no-op and a conflicting row fails closed.
- Data-contract unlock requires the pending migration row plus exact direct-pair absence, complete owner/grant observation, named membership paths/group roots, effective-DML/self-grant truth, no indeterminate statement, mode gate, verifier/time, and canonical composite digest. In separated mode the different verifier observes it; combined records no independent observer. Recompute before every App shell, bound-unshared, user-access, and trust-generation plan. Neither the pending row nor a runtime-trust candidate unlocks runtime.
- Treat account/workspace/metastore administrators, customer object/attestation/trust-ledger owners, every granted group, the seal-verifier/trust-committer group, Bundle deployment principals, App/Job managers, and all accepted Service Principal User/Manager principals as explicit roots. `MANAGE` is self-grant capability, not current data access. The v1 verifier must also be the account administrator who compares each native Permissions roster with the signed expected roster; record that component as administrator-attested, point-in-time, and non-independent, never machine-verified or continuous. If policy requires a separate identity reviewer, administrator-resistant evidence, or API-current custody, fail readiness.
- Enumerate steady-state runtime DML exactly. In `RUNTIME_DML_ONLY`, the collector writes only three normalized evidence tables; optional enrichment only three snapshots; the P6 App only `pending_identity_requests`, `action_ledger`, and the singleton `runtime_authorization_fence`; and the role Job only seven named identity/action tables. A `BOOTSTRAP_ALLOWED` invocation is a separately surfaced installation phase, not ordinary collection, and its closed DDL manifest and postconditions are reviewed independently. Runtime principals get status-view `SELECT` as needed but no migration- or trust-ledger write. Only the verifier/trust committer has persistent `SELECT`+`MODIFY` on `runtime_trust_ledger`; P6 additionally gives that verifier fence `SELECT`+`MODIFY` and action-ledger `SELECT` for drain/recovery/reopen. It has no migration/evidence/identity/enrichment DML. Every unlisted runtime principal is denied both ledger writes, fence DML, ownership, `MANAGE`, and schema-level DDL grants. Table-level `MODIFY` still permits some schema alteration and must be disclosed rather than described as no DDL authority. App code/managers and the verifier are explicit P6 fence/Run Now trusted roots, not administrator-resistant separation.
- Persist trust only in customer-security-owned `runtime_trust_ledger` plus sanitized `runtime_trust_status_v`. One physical Delta row is one logical event; each generation freezes one canonical sorted, unique component array containing `BASE_OBSERVABILITY` and only the installed optional components. Fixed marked idempotent events are `MANIFEST_REGISTERED`, `TRUST_CANDIDATE`, `SNAPSHOT_ACCEPTED`, or `SNAPSHOT_INVALIDATED`; missing, partial, duplicate, conflicting, replayed, unreadable, invalidated, or superseded generations fail closed. Use the exhaustive DDL, four-event nullability matrix, literal RFC 8785/SHA-256 domain objects, dependency order, and Python/SQL/JavaScript golden vectors in the product plan—never a shorthand digest group. Delta integer columns remain numeric, but every canonical JSON integral property uses the versioned unsigned decimal-string encoding and exact range; never serialize a trust integer as a JSON number. Candidate and acceptance retain distinct pre-start `STOPPED` and post-start `ACTIVE` observation digests/times while requiring an identical stable graph, selected deployment, and zero pending deployments. `statement_evaluated_at` comes from one server `current_timestamp()` at query-evaluation start and is never called a commit time. Reuse only the original self-anchored roster event from a complete accepted, uninvalidated source chain; never reference an intermediate candidate or move the original anchor. Optional client signatures are non-authoritative audit metadata and never affect the status view or a runtime gate. App/writers read only the one-row-per-installation latest-generation summary, require exactly one matching component element, stamp the accepted snapshot, and never promote it; base collection may continue stale with observation-time provenance.
- Treat collector trust as observation-time provenance only. The first successful `dbt_artifact_registry` AttemptKey-root write freezes the accepted snapshot, generation, state, and status-view query-evaluation time. Retry, repair, reconciliation, and later evidence rows reuse that tuple; it never enters AttemptKey, artifact/log hashes, pair validation, capture precedence, or Lakeflow/dbt/node/test outcomes. A later trust change may commit before the evidence row without rewriting the stamp. Label **Trust observed by collector** separately from **Current runtime trust** and never claim trust at evidence commit.
- P6 must use the exact unpartitioned Serializable singleton fence contract in the product plan. Install it only through the marked sequence: managed Delta create with explicit `UTF8_BINARY`; separate isolation-property operation; eleven separate named `ALTER TABLE ... ADD CONSTRAINT` operations; complete constraint/property/shape readback; then the one `CLOSED`/`INSTALL` row. Inline Delta `CHECK` clauses in `CREATE TABLE` are prohibited. Every operation has its own marker/digest/pre/post-state, and missing/subset/conflicting/non-empty/wrong-collation/property/already-initialized partial states resume or block exactly; never trust a constraint name without its expression. Every claim, drain, phase update, takeover, release, reopen, and retirement then conditionally modifies the same verified row/version and reads it back. The claim commit authorizes one action; the drain commit blocks later admissions; an earlier claim may still need same-token Jobs reconciliation and terminal cancellation/wait. Use the current `status.state`/termination-details projection first and deprecated `state` only when current status is absent; `INTERNAL_ERROR` is terminal failure, while unknown/conflicting/deleted/denied/incomplete evidence remains indeterminate. No trust append, App stop/binding change, refresh, upgrade, rollback, stop, or uninstall mutation starts before the matching drain is exactly `CLOSED` after terminal ledger plus final run/task/compute inventory. Acceptance never reopens automatically. Lease expiry only enables reconciled takeover; it never closes, releases, reopens, or proves quiescence. Never call the fence atomic with Jobs, say `DRAINING` means no run can launch, claim it blocks external schedules/callers, or claim it protects against a compromised App/admin root.
- For a fresh install, first apply a reviewed Direct App-shell plan with no product data/warehouse/Job/secret/Volume binding, no user `CAN_USE`, `lifecycle.started:false`, and no deployment. Next generate and separately approve a fresh bound-unshared plan: it may add only the exact reviewed `config.env[].value_from` resource-key mappings plus their read-only resource bindings, keeps user permissions empty, and remains stopped and undeployed. Direct App Update applies the resources but does not retrofit deploy-only environment configuration; the subsequent pinned `bundle run` consumes the current bound overlay and creates the only deployment. Require an empty deployment baseline, invoke it once without a wrapper retry, stop on every success/error path, paginate again, and accept only exactly one new terminal successful matching `SNAPSHOT`; zero, multiple, pending, auto-sync, mismatched, or unaccounted deployments fail closed. The pinned runner starts compute before deploying and may internally reissue its deploy POST, so disclose the bounded billed interval and the App principal/code, deployment principal, App managers, workspace administrators, and Databricks control plane as trusted roots. During that interval the App has only the reviewed warehouse and sanitized-view bindings, no user access, and no accepted runtime trust; call it bound and fail-closed, never zero-authority. After reconciliation and `STOPPED`, generate a third fresh plan whose only delta is the approved group `CAN_USE`; source/code/build/artifact/command/configuration/resources/deployment stay unchanged. Perform pre-start GA observation plus native roster review and append only a candidate. Use explicit `databricks apps start <name>` to start that same deployment, reobserve the same ID after `ACTIVE`, and only then append acceptance/unlock. Prohibit no-argument/bare `apps deploy` throughout the wrapper, and prohibit `bundle run` or `lifecycle.started:true` after the one bound deployment invocation. An existing App with any deployment is not a fresh-install shell and must block or use a separately qualified upgrade path. Mismatch or commit ambiguity remains setup-only and stops by default.
- Show the pre-start transition-audit time separately from the post-start machine-freshness and original-roster-freshness times, their two validity ages, oldest validity-component age, expiry, non-independent qualifier, and running-cost state. An early unchanged refresh may reuse only an eligible original self-anchored roster observation without extending it; a 24-hour refresh repeats native review. Never imply point-in-time observation stops a malicious named root, detects later change immediately, or follows from the migration seal.
- Do not add a Jobs-visible receipt carrier in v1. The returning actor reads the sanitized Delta attestation view and recomputes the live composite acceptance tuple directly; Jobs runs/output expire after 60 days and add no authority.
- Keep optional system-table enrichment disabled and absent from the base manifest. Its separate enablement uses one paused fixed-code Job and dedicated principal, reads only accepted `system.lakeflow` and `system.billing` schemas, and filters immediately through one product-owned `(workspace_id, job_id)` scope table. The attended fixed U2M data plane is the only supported scope writer through an approved idempotent DML-only scope-update envelope. Lakeflow exposure is account-wide within the current cloud region; billing exposure is account-global. The enrichment Job writes only `lakeflow_job_run_snapshot`, `lakeflow_dbt_task_run_snapshot`, and `dbt_job_cost_snapshot`, has no DDL, and never copies raw identity or unonboarded/cross-workspace rows. The App never receives `system` access.
- Use `Personal Data`, not `PII`, in product documentation and UI text.

## Evidence and security

- Treat dbt artifacts and logs as sensitive. They can contain SQL, relation names, variables, metadata, database messages, and user-controlled text.
- Normalize an allowlist of fields. Do not expose raw SQL, raw log messages, arbitrary metadata, environment values, or full artifacts in the ordinary App UI.
- Treat `X-Forwarded-User` as an IdP-provided value that may be an email, not as a documented opaque Databricks principal ID. Keep the raw value request-local and use only the reviewed pseudonymous actor-enrollment contract for controlled actions.
- Never put an enrollment credential, HMAC key/input, browser credential, token, raw forwarded identity, or other secret in a URL, shell argument, Job parameter, checkpoint, log, screenshot, or review fixture.
- Never commit credentials, tokens, workspace URLs, account identifiers, emails, signed artifact URLs, or captured customer evidence.
- Pin runtime dependencies and record the tested dbt Core, dbt-databricks, Python, and Databricks CLI compatibility matrix.
- Disable dbt anonymous usage and artifact-ingest upload explicitly in every generated dbt command.
- Mark every Databricks feature as GA, Preview, Beta, or Experimental from a current official source. Preview features must be optional unless a reviewed decision explicitly accepts them.

## Required review ownership

Every independently deliverable part must receive the review routed by
`docs/plans/review-process.md`: one dominant domain owner for an ordinary slice,
the named second owner for a boundary trigger, and all three owners only for the
contract-freeze, local-release-candidate, and live-Azure-proof milestones. This
keeps Databricks platform, dbt Core, and usability accountability continuous
without repeating three source-wide audits for every edit.

Review follows dependency order rather than majority vote. The owner of a
technical fact freezes the supported mechanism before downstream dbt or
usability effects are reviewed. A boundary reviewer may reject the impact in
their own domain but must not prescribe an unsupported mechanism in another
owner's domain. Intermediate support-contract edits route only to owners whose
assumptions changed; all three review the consolidated packet once at contract
freeze.

A `CHANGES_REQUIRED` or `BLOCKER` verdict from any required owner prevents that
slice or milestone from completing. Resolve the finding in the implementation
or record an explicit accepted-risk decision, then request re-review only for
the changed criterion and any newly crossed boundary.

Documentation also requires separate Diataxis information-architecture, FastAPI-style readability, security/compliance, and documentation usability/accessibility reviews, followed by rendered-site validation when pages exist. Review each file or section as it is produced; do not batch an entire documentation set into one final review.

## Implementation cadence

- Work in small, independently testable slices.
- Keep domain and parsing logic outside UI handlers so it can be unit tested without Databricks.
- Run unit tests on every change, `databricks bundle validate` for Bundle changes, and a bounded staging integration test before production promotion.
- Python changes must pass `ruff` and `ty` in addition to relevant tests.
- Do not start paid cloud resources during planning or static validation.
- Any live test must declare its cost boundary, create no schedule by default, and finish with an inventory proving Apps and warehouses are stopped and temporary runs/resources are absent or removed.

## Source policy

- Prefer current official Databricks and dbt documentation and first-party GitHub repositories.
- Record decisions and dated source links in `docs/research/source-register.md`.
- Do not turn a roadmap item, issue, release candidate, Preview, Beta, or Experimental feature into a production claim.
